Milestone M1 — USB-only Bluetooth bring-up

Last verified: 2026-07-18 · Executed — PASS · Repo state: the plan below has been run bench-side against the single physical board. The board is flashed with the Idle-output a2dp_sink_stream build (source-unmodified; only the Idle Output menuconfig choice + device name were set) and was left running it after a PASS. No repo firmware/wiring commit was made by this bring-up — see §9 for the run record and evidence.

A single-session, bench-usable plan to prove one thing: the ESP32 T-Display advertises as a Bluetooth speaker, a phone discovers and connects to it, and the serial monitor shows the A2DP audio-stream state — using nothing but the board and a USB cable. It is the detailed expansion of milestone M1 / Phase 3 in the main build plan.

The PCM5102A DAC is not needed for M1. Leave it disconnected and unpowered. M1 is a transport proof, not an audio-output test. The reference firmware is configured to discard the decoded audio, so no I2S pins, no DAC, no amplifier, and no speaker are involved. Everything about sound begins at Phase 4 / M2.

1. Objective, definition of done, and non-goals

Objective
Flash Espressif's pinned reference A2DP-sink example — configured to discard audio — and prove the Bluetooth link end to end: the device is discoverable, a phone pairs and connects, and the serial log shows the A2DP connection and audio-stream state.
Milestone
M1 (dashboard · Phase 3). Delivers the “ESP32 advertises as a Bluetooth speaker” gate.
Definition of done
All of: (a) the monitor shows the Bluetooth stack initialise and A2DP init complete; (b) the phone lists the device by name and pairs; (c) the monitor logs A2DP connection state: Connected with the phone's MAC; (d) starting playback logs A2DP audio state: Started and a rising Audio packet count; (e) a clean disconnect and reconnect are observed; (f) the evidence in §9 is captured. No sound is expected or required.
Non-goals
Audible output, any I2S/DAC/amp/speaker work, GPIO configuration, display bring-up, AVRCP button control, power/enclosure work, and opening the boombox. These are later milestones and must not be attempted here.
What passing does not mean
M1 has been run and passed (§9). Passing proves the transport only; it does not mean any audio path works — that is M2.

5. Safety & go/no-go preflight

Work top to bottom. Any NO-GO stops the session until it is resolved — do not improvise past it.

  • Device identity & port. Board plugged in via a direct data cable. Rediscover the port: ls /dev/cu.usbserial* /dev/cu.wchusbserial*. Confirm it is the CH9102 bridge. Record the exact port string. NO-GO if no port enumerates.
  • Toolchain health. scripts/doctor.sh exits environment ready (ESP-IDF v6.0.2 present, tag matches, port visible). NO-GO on any FAIL line.
  • Backup verified. Confirm the recorded pre-write backup still hashes to its logged SHA-256 (shasum -a 256 <backup> vs the record in §16). Do not take a new “first” backup — one already exists. NO-GO if the hash does not match.
  • No bulk erase. The plan uses only idf.py flash, which does not bulk-erase. erase-flash is forbidden without explicit per-session user approval — it is not part of M1.
  • No GPIO configuration. The build must select the Idle audio output (§6) so that no I2S or DAC pin is configured. Confirm the output choice before flashing. NO-GO if the build is set to External I2S or Internal DAC.
  • Power & USB conditions. Direct cable, no hub; laptop on stable power; nothing else wired to the board headers.
  • Rollback artifact ready. Know both rollback paths before flashing: reflash the repo smoke app, or restore the factory image from backup (§11).

6. Firmware strategy & configuration delta

The strategy is deliberately the least invasive path that still exercises the whole Bluetooth transport: run Espressif's stock example unmodified and select its built-in “discard” output. No source edit is required, and no GPIO is configured.

6.1 Verified source behaviour (ESP-IDF v6.0.2)

The following are read directly from the checked-out pinned tree at examples/bluetooth/bluedroid/classic_bt/a2dp_sink_stream and its shared classic_bt/common components (exact files in §14). Each is VERIFIED in source, not inferred:

  • The example advertises as a Classic-Bluetooth A2DP sink. The device name is CONFIG_EXAMPLE_LOCAL_DEVICE_NAME, default ESP_SPEAKER.
  • At stack-up it registers GAP + A2DP callbacks, calls esp_a2d_sink_init(), and sets scan mode to ESP_BT_CONNECTABLE, ESP_BT_GENERAL_DISCOVERABLE — so a phone can discover and connect. On connect it becomes non-discoverable; on disconnect it returns to connectable/discoverable.
  • Audio output is a build-time choice, EXAMPLE_A2DP_SINK_OUTPUT, with three options: External I2S Codec (the default — drives GPIO), Internal DAC, and Idle Output, whose help text reads literally “Select this to discard a2dp sink data.”
  • The Idle output implementation is a set of no-ops. It includes no I2S, GPIO, or DAC driver and touches no pin. The three I2S pin settings (EXAMPLE_I2S_LRCK_PIN=27, _BCK_PIN=26, _DATA_PIN=25) depend on the External-I2S choice, so in Idle mode they do not even exist in the configuration.
  • Even while discarding audio, the internal-codec data handler still logs Audio packet count: N every 100 packets — decoded PCM is flowing, proving the stream without a DAC. The example README confirms this: without a codec “the example will only show a count of audio data packets received silently.”
  • Connection and stream state are logged by the A2DP event handlers: A2DP connection state: {Disconnected|Connecting|Connected|Disconnecting}, [MAC], A2DP audio state: {Suspended|Started}, A2DP PROF STATE: Init Complete, and the protocol/capability lines. These are the M1 evidence.
  • The committed sdkconfig.defaults already enables the right stack: CONFIG_BT_ENABLED=y, CONFIG_BTDM_CTRL_MODE_BR_EDR_ONLY=y (controller in Classic BR/EDR mode, no BLE), CONFIG_BT_BLUEDROID_ENABLED=y, CONFIG_BT_CLASSIC_ENABLED=y, and CONFIG_BT_A2DP_ENABLE=y.
  • Documented limitations (README): SBC codec only, decoded to 44.1 kHz stereo 16-bit PCM; at most one A2DP source connection at a time; A2DP sink cannot run together with A2DP source.

6.2 Proposed configuration (no source change)

The only decisions are menuconfig selections — clearly separated here from the verified facts above so a reviewer can see what is stock versus what this plan chooses:

  • CHOICE Set the audio output to Idle Output (CONFIG_EXAMPLE_A2DP_SINK_OUTPUT_IDLE=y). This is the whole adaptation: it flips the stock default away from External I2S so that zero GPIO is configured. No C code is edited.
  • CHOICE Optionally set an identifiable device name, e.g. CONFIG_EXAMPLE_LOCAL_DEVICE_NAME="RX-5235 Digital Tape" (matching Phase 3). The stock ESP_SPEAKER is equally valid — record whichever is flashed.
  • Everything else stays at the example's committed defaults.

If a reviewer prefers not to rely on the Idle choice at all, the equivalent minimal source adaptation would be to leave the output at Internal-codec and make audio_sink_srv_data_output() a no-op — but that is unnecessary, because the stock Idle option already is exactly that. Prefer the stock option; do not modify source.

7. Preparation & command sequence

Read-only and build steps are safe to repeat. Exactly one step writes to the board, and it is called out. All commands run in the pinned environment; the bench project lives outside this repository until §15 folds any keeper code into components. <port> is the only placeholder — discover it, never guess it.

7.1 Read-only preparation

# In the boombox repo checkout — confirm the pinned toolchain is healthy
scripts/doctor.sh

# Discover the serial port (an observation each session, not a constant)
ls /dev/cu.usbserial* /dev/cu.wchusbserial*
# -> record the exact string, e.g. /dev/cu.usbserial-XXXXXXXXXXX

# Re-verify the existing full-flash backup against its recorded hash
shasum -a 256 "$HOME/Library/Application Support/Boombox/backups/<backup-file>"
# -> must equal the SHA-256 recorded in build plan section 16

7.2 Instantiate & build the example (no writes to the board)

# Enter the pinned ESP-IDF environment (as scripts/build.sh does internally)
source "$HOME/esp/esp-idf-v6.0.2/export.sh"

# Create a bench project from the pinned example, OUTSIDE the repo
cd ~/bench            # any scratch dir outside the boombox repo
idf.py create-project-from-example \
  "bluetooth/bluedroid/classic_bt/a2dp_sink_stream"
cd a2dp_sink_stream

idf.py set-target esp32

# Select the Idle output (and optionally the device name).
# Menu path: "A2DP Sink Internal Codec Example Configuration"
#   -> "A2DP Sink Output" -> "Idle Output"
# and (optional) "A2DP Example Configuration" -> "Local Device Name".
idf.py menuconfig
# Equivalent non-interactive form — append to sdkconfig.defaults, then build:
#   CONFIG_EXAMPLE_A2DP_SINK_OUTPUT_IDLE=y
#   CONFIG_EXAMPLE_LOCAL_DEVICE_NAME="RX-5235 Digital Tape"

# Build only — this does not touch the board
idf.py build

# Capture the config hash for the evidence log
shasum -a 256 sdkconfig

Before the write step, re-confirm in menuconfig (or sdkconfig) that Idle Output is selected and that no EXAMPLE_I2S_*_PIN option is present. This is the last gate that keeps M1 at zero GPIO.

7.3 The one deliberate flash + monitor (writes to the board)

This is the only step that changes the board. It overwrites the smoke app (already backed up). A plain flash does not bulk-erase — keep it that way.

# Deliberate, explicit flash over the direct cable, then open the monitor
idf.py -p <port> flash monitor
# Monitor exits with Ctrl-]  (console is 115200 8N1 on UART0)

If long transfers stall on the CH9102 bridge, add -b 230400 (idf.py -p <port> -b 230400 flash monitor).

8. Bench procedure

Do these in order, one variable at a time, watching the monitor throughout. Audio need not be audible — the log is the proof.

  1. Boot & stack. On flash+monitor, watch for the boot banner, then Bluetooth init and A2DP PROF STATE: Init Complete. The device is now discoverable.
  2. Discovery. On the phone, open Bluetooth settings and scan. The device name (RX-5235 Digital Tape or ESP_SPEAKER) should appear within ~10 s.
  3. Pair & connect. Tap to pair. Pairing uses Secure Simple Pairing by default (typically no PIN prompt on a modern phone; a legacy fixed PIN 1234 is the fallback). The monitor should log A2DP connection state: Connecting then Connected, [phone-MAC].
  4. Stream start. Play audio from a music app. Watch for A2DP audio state: Started followed by Audio packet count: 100, 200, 300… climbing. That rising count is the proof that decoded audio packets flow — with no DAC attached.
  5. Stream stop. Pause playback; expect A2DP audio state: Suspended and the packet count to stop advancing.
  6. Reconnect. Disconnect from the phone (or power-cycle the board): expect A2DP connection state: Disconnected and a return to discoverable. Reconnect and confirm the connected log again.
  7. Negative / retry checks. (a) With the board disconnected, confirm it is discoverable again from a fresh scan. (b) Forget the device on the phone and re-pair from scratch. (c) Optionally confirm defined behaviour when a second phone tries to connect while one is connected (single source expected).

This mirrors the plan's §11.1 Bluetooth test matrix (Discovery, Pairing, Reconnect, Second-device) — run those rows here and record them in §9.

Do not start changing GPIOs. If M1 will not pass, the fix is in the build config, the cable/port, or the phone — never in wiring a pin. There is nothing to wire in M1.

11. Stop conditions & rollback

Stop the session if: pairing fails repeatedly after the troubleshooting branches, the BT stack will not start after a clean rebuild/reflash, or anything tempts you toward wiring or bulk-erasing. Capture the monitor output first, then roll back.

Rollback options (either returns the board to a known state; never bulk-erase):

  • Reflash the repo smoke app — rebuild the boombox repo app and idf.py -p <port> flash it back. Fastest way to a known-good, board-neutral state.
  • Restore the factory image from the preserved backup, following plan §16. Use this if the smoke app itself is in doubt.

Both leave the DAC untouched (it was never connected) and require no erase. Record what happened on the M1 bead either way.

  • Powered DAC config-pin defaults (FLT / DEMP / XSMT / FMT) and on-board SCK-to-GND — measured with the module powered.

None of the above is part of M1. The M1 build's zero-GPIO stance is exactly what lets M1 run before that pin/DAC work is finished.

Comfortably one bench session; no waiting on external parts (the DAC is not involved).
Natural pause points
After the read-only preflight (§7.1); after a green build but before the deliberate flash (§7.3); after capturing the connected + packet-count evidence (§9).
Can be completed in one sitting?
Yes — everything here is a single USB-only session.
